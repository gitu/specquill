# Tenant-scoped URLs and a first-class config tenant

Tenants SHALL be completely separate: named in every URL, resolved from
the path alone, and uniform across providers — a GitHub organisation, a
GitHub user account, and an on-prem installation are the same kind of
thing.

> **REQ-022.1 · MUST** — Every tenant-scoped API route SHALL live under
> `/api/t/{tenant}/…` and every tenant-scoped SPA route under
> `/t/{tenant}/…` — websockets and raw asset URLs included. No header,
> query parameter or client-side pin SHALL select the tenant. Global
> routes are exactly: identity (`/api/me`, `/auth/*`), share downloads
> (`/share/*`) and webhooks (`/hooks/*`).

> **REQ-022.2 · MUST** — The path tenant SHALL be authorized on every
> request: an unknown slug yields 404; a known tenant without caller
> visibility (membership or at least one repo grant) yields 403. Data
> cached client-side SHALL be keyed by tenant so nothing leaks across a
> switch.

> **REQ-022.3 · MUST** — Switching tenants SHALL be plain navigation — no
> reload, no carried project/branch state; `/` resolves to the last-used
> tenant, else the sole membership, else a tenant picker.

> **REQ-022.4 · MUST** — The on-prem installation SHALL be an ordinary
> `config`-provider tenant declared by a YAML `tenant:` block (slug,
> display name, `default_role`, `admin_emails`); when the block is omitted
> but projects are configured, a default block is synthesized. No code
> path outside config normalization SHALL know a hardcoded tenant slug.

> **REQ-022.5 · MUST** — A hosted deployment SHALL boot with zero
> configured projects and no config tenant — tenants then exist solely
> through GitHub App installations.
