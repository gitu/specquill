# Four-level repository roles with a protected-merge gate

Repository permissions SHALL form a four-level ladder — `viewer < editor <
maintainer < admin` — separating the ability to write from the ability to
merge into a protected branch, and repo administration from tenant
administration.

> **REQ-021.1 · MUST** — Every repository route SHALL be gated by one
> effective role per (user, repo): `viewer` reads and comments; `editor`
> additionally writes workspace branches, commits, opens/approves/closes
> PRs, co-edits and uses the copilot; `maintainer` additionally merges PRs
> into protected branches and manages share links; `admin` additionally
> manages the repository's grants and settings.

> **REQ-021.2 · MUST** — Merging a pull request into a protected branch
> SHALL require `maintainer`. A PR targeting an unprotected branch merges
> at `editor`. Approval and merge are distinct capabilities — an editor's
> approval SHALL NOT enable the merge.

> **REQ-021.3 · MUST** — GitHub-derived roles SHALL map onto the ladder
> per repository: `pull` → viewer, `triage`/`push` → editor, `maintain` →
> maintainer, `admin` → admin. The effective role is the maximum of the
> derived role and any explicit grant ([REQ-020](REQ-020.md)).

> **REQ-021.4 · MUST** — Repo `admin` SHALL be distinct from tenant admin:
> a tenant admin derives repo admin on every repository, but a repo admin
> (derived or granted) holds no tenant management rights. Managing a
> repository's grants requires repo admin, not tenant admin.

> **REQ-021.5 · MUST** — UI affordances SHALL follow the same ladder: no
> write chrome below `editor`, no merge control below `maintainer`, no
> grant management below `admin` — and gates default closed until the role
> is known.
