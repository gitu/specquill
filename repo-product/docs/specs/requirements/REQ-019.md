# GitHub-App tenants with derived authorization

A GitHub App installation SHALL be a tenant: its repositories are the
tenant's candidate workspaces, and a user's rights in the tenant are their
rights on GitHub — derived by default, never duplicated. Explicit
per-repository grants ([REQ-020](REQ-020.md)) layer on top of the derived
roles and are never overwritten by a sync.

> **REQ-019.1 · MUST** — Installing the app SHALL create the tenant;
> uninstalling or suspending SHALL revoke every membership immediately.
> Tenant state survives a re-install.

> **REQ-019.2 · MUST** — Roles SHALL derive from GitHub repo permissions
> (`admin` → admin, `write` → member, `read` → viewer), synced on demand
> with a bounded cache. Repo access is derived per repository; the
> tenant-level role is the maximum across the tenant's repos and gates
> tenant management (REQ-020.3). A sync failure SHALL never revoke existing
> access mid-session.

> **REQ-019.3 · MUST** — On a fresh installation with no adopted repos, an
> account admin SHALL still bootstrap as tenant admin (derived from the
> installation's candidate repositories) so the repo picker is reachable.

> **REQ-019.4 · MUST** — A tenant admin SHALL pick which installation
> repositories become workspaces (writable projects) or reference sources;
> the picker SHALL never reach repositories the installation does not grant.

> **REQ-019.5 · MUST** — Git operations for a github tenant SHALL
> authenticate with short-lived installation tokens scoped to that
> installation — no shared credentials, tokens never cross tenants.

> **REQ-019.6 · MUST** — Users with multiple memberships SHALL choose their
> active tenant; every API request is evaluated within exactly one tenant.
